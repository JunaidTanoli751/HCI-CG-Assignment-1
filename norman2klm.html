HCI Analysis Frameworks

Introduction: Why We Need Analysis Models

Imagine you're designing a new banking app for Pakistan. How do you know if your design is good? How can you predict if users will struggle? How do you compare two different design approaches objectively?

This is where HCI analysis models come in. They provide structured, scientific methods to evaluate and predict user interaction with systems. Three fundamental frameworks form the foundation of HCI analysis:

  • Norman's Design Principles ‚Äî Qualitative guidelines for good design
  • GOMS Model ‚Äî Task analysis framework for expert users
  • KLM (Keystroke-Level Model) ‚Äî Quantitative time prediction method

Part 1: Norman's Seven Principles of Design

Don Norman identified seven fundamental principles that make interfaces intuitive and usable. Let's explore each with Pakistani examples.

Principle 1: Discoverability

"Can users figure out what actions are possible?"

Definition: Users should be able to discover available actions without instruction or prior training.

Pakistani Example: ATM Machine
Good Discoverability Poor Discoverability
HBL ATM: Labeled buttons next to screen options
"Cash Withdrawal" ‚Üí [Button 1]
"Balance Inquiry" ‚Üí [Button 2]
Clear numbered buttons match screen positions 		Old MCB ATM: Unlabeled function keys
User must guess which F-key corresponds to which option
No visual mapping between screen and buttons
Design Checklist for Discoverability:
  • Are interactive elements visually distinct from static content?
  • Can users identify all possible actions without help documentation?
  • Do icons have text labels for clarity?
  • Is navigation structure visible and comprehensible?

Principle 2: Feedback

"Does the system communicate what happened after user actions?"

Definition: The system should provide immediate, clear information about the result of every action.

Real Example: Mobile Banking

  2. Success screen with green checkmark and sound
  3. "Rs 1,000 sent to 03XX-XXXXXXX"
  4. Transaction ID: ABC123456
  5. New balance displayed
  6. Option to share receipt
‚ùå Poor Feedback (Old NADRA system)
Action: Submit online application
Feedback:
  • Loading... (no indication of progress)
  • Long wait (30+ seconds)
  • Generic "Application submitted" message
  • No confirmation number
  • No indication of next steps or timeline

Principle 3: Conceptual Model

"Does the design help users build an accurate mental model?"

Definition: The conceptual model is the user's understanding of how the system works. Good design makes this model accurate and easy to form.

Example: Online Shopping Cart Metaphor

Why Daraz uses a "cart" icon instead of "basket" or "bag":

  • ‚úÖ Supermarket carts are familiar to all Pakistanis
  • ‚úÖ You can add/remove items before checkout (matches real cart)
  • ‚úÖ Cart "fills up" as you add items (visual parallel)
  • ‚úÖ You can "empty" the cart (delete all) ‚Äî real-world action

Mental Model Mismatch Example:

System Design User's Mental Model Actual Behavior Result
Pakistan Post tracking "consignment number" Like courier tracking: Enter number, see real-time location Shows only major checkpoints, updates once daily Confusion, frustration, support calls
Careem "Schedule Ride" feature Book ride in advance, captain assigned at booking time Captain only assigned 15 mins before pickup time Users concerned ride won't be available

Principle 4: Affordances

"What actions does the design suggest are possible?"

Definition: Affordances are perceived properties that suggest how an object should be used.

Physical Affordances in Pakistan:
  1. Metro Bus Door Buttons: Large, raised button affords pushing. Green color when active affords "safe to press."
  2. Motorcycle Helmet Strap: Two D-rings afford threading one through the other (though many users struggle initially).
  3. Tetrapak Juice Box: Perforated circle affords pushing straw through. Poor affordance: users often push wrong side.
Digital Affordances:
Element Visual Cue Afforded Action
Underlined blue text Color + underline Clickable link
Button with shadow/gradient 3D appearance Push/tap action
Slider with draggable thumb Raised handle on track Drag left/right
Text field with blinking cursor Cursor animation Type text here
False Affordances (Design Anti-patterns):
  • ‚ùå Non-clickable text styled like links (underlined, blue)
  • ‚ùå Disabled buttons that look exactly like enabled ones
  • ‚ùå Static images that look like buttons (drop shadow, hover effects)

Principle 5: Signifiers

"Where exactly should users act?"

Definition: Signifiers indicate where the action should take place. They make affordances perceivable.

Difference: Affordances vs Signifiers
Understanding the Distinction
Object Affordance Signifier
Door Can be pushed or pulled "PUSH" sign / vertical handle (pull) vs flat plate (push)
Touchscreen Can be touched anywhere Button graphics, highlights showing touch areas
Form Fields can receive text Labels, placeholder text, required field asterisks (*)

Principle 7: Constraints

"What limitations guide correct behavior?"

Definition: Constraints limit possible actions, reducing errors and guiding users toward correct behavior.

Four Types of Constraints:
Constraint Categories with Pakistani Examples
Type Description Example
Physical Physical properties prevent incorrect actions ‚Ä¢ USB-C cable only fits one way
‚Ä¢ SIM card notched corner ensures correct orientation
‚Ä¢ ATM card slot shaped to accept card only one way
Semantic Meaning of situation limits possible actions ‚Ä¢ Can't send money from empty JazzCash wallet
‚Ä¢ Can't book Careem ride to current location
‚Ä¢ Can't order from closed restaurant on Foodpanda
Cultural Cultural conventions constrain behavior ‚Ä¢ Reading Urdu right-to-left constrains text layout
‚Ä¢ Iftar time shown in 12-hour format (not 24-hour)
‚Ä¢ Honorifics in contact forms (Mr./Ms./Dr.)
Logical Reasoning about relationships limits actions ‚Ä¢ Can't checkout empty shopping cart (no items = nothing to buy)
‚Ä¢ Can't rate driver before ride completion
‚Ä¢ Can't withdraw more than account balance
Constraints in Form Design:

Good form design uses constraints to prevent errors:

  • ‚úÖ Phone number field accepts only digits (physical constraint)
  • ‚úÖ Email field requires "@" symbol (logical constraint)
  • ‚úÖ Date picker prevents selecting past dates for appointments (semantic constraint)
  • ‚úÖ Required fields marked with * (cultural/conventional constraint)

Part 2: GOMS Model

GOMS stands for Goals, Operators, Methods, and Selection rules. It's a framework for analyzing task completion by expert users.

GOMS Components Explained:

Goals
What the user wants to accomplish. Can be hierarchical (main goal ‚Üí sub-goals).
Operators
Basic actions user can perform. Examples: click mouse, press key, read text.
Methods
Sequences of operators to accomplish goals. Multiple methods can achieve same goal.
Selection Rules
Conditions determining which method to use in a given situation.

Practical Example: Transferring Money via JazzCash

Goal Hierarchy:
Main Goal: Transfer Rs 5,000 to friend

Sub-Goals:
‚îú‚îÄ 1. Open JazzCash app
‚îú‚îÄ 2. Authenticate (enter PIN)
‚îú‚îÄ 3. Navigate to send money function
‚îú‚îÄ 4. Enter recipient details
‚îú‚îÄ 5. Enter amount
‚îú‚îÄ 6. Review transaction
‚îî‚îÄ 7. Confirm transfer
Methods Available:
Different Methods to Send Money
Method Operators Time Prerequisite
Method 1: Favorites 1. Tap "Send Money"
2. Select from favorites list
3. Enter amount
4. Confirm PIN 		~15 sec Recipient already in favorites
Method 2: Recent 1. Tap "Send Money"
2. Select from recent
3. Enter amount
4. Confirm PIN 		~18 sec Sent money to this person recently
Method 3: Manual Entry 1. Tap "Send Money"
2. Enter 11-digit mobile number
3. Enter amount

Selection Rules:
  1. IF recipient is in favorites ‚Üí Use Method 1
  2. ELSE IF sent money to this recipient in last 7 days ‚Üí Use Method 2
  3. ELSE ‚Üí Use Method 3 (manual entry)
GOMS Analysis Insights:

This GOMS breakdown reveals:

Part 3: KLM (Keystroke-Level Model)

KLM is a simplified GOMS variant that predicts task time by breaking interactions into primitive operations and summing their durations.

KLM Operators and Standard Times:

Standard KLM Operator Times (Card et al., 1983)
Operator Symbol Description Time (seconds)
Keystroke K Press and release single key 0.20
Pointing P Move mouse pointer to target 1.10
Button B Press or release mouse button 0.10
Homing H Move hands between keyboard/mouse 0.40
Mental Act M Mental preparation for action 1.35
System Response R(t) Wait for system to respond Variable

Task: Pay K-Electric bill on bank website

Conclusion: Combining the Models

Norman's Principles, GOMS, and KLM work together to provide comprehensive UX evaluation:

  • Norman's Principles identify what's wrong qualitatively
  • GOMS structures how users accomplish tasks
  • KLM quantifies how long tasks take

For Pakistani developers, these tools help create interfaces that work for our unique context‚Äîfrom multilingual users to varying internet speeds to diverse technical literacy levels.

"Good design begins with understanding your users. These models give us structured ways to understand, analyze, and improve. But remember: models are tools, not rules. Always validate with real user testing."
